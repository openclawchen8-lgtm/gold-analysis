# ADR-001 - Gold Analysis 多專案架構設計

**日期**: 2026-04-10  
**決策者**: 團隊（寶寶、碼農1號、碼農2號、安安、樂樂）  
**狀態**: ✅ 已接受

## 背景

隨著 gold-analysis 專案發展，功能需求逐漸分層：
- **core**: 核心分析引擎（已完成）
- **extend**: 擴展功能（投資組合、告警、回測）
- **advanced**: 進階功能（ML、實盤交易）
- **platform**: 平台功能（API、社區、移動端）

需要決定如何組織這些功能模組。

## 考慮選項

### 選項A: 單一專案，功能分目錄
| 優點 | 缺點 |
|------|------|
| 簡單，一個 repo | 代碼膨脹，編譯慢 |
| 統一版本管理 | 耦合度高 |
| 部署簡單 | 無法獨立發布 |

### 選項B: 完全獨立專案（4個 repos）
| 優點 | 缺點 |
|------|------|
| 完全解耦 | 管理複雜 |
| 可獨立發布 | 版本同步困難 |
| 團隊分工清晰 | 跨專案依賴難處理 |

### 選項C: 單一 Repo，多專案目錄（採用）
| 優點 | 缺點 |
|------|------|
| 統一版本管理 | 需要良好目錄規劃 |
| 跨專案依賴容易 | repo 較大 |
| 團隊協作方便 | |
| 可分階段開發 | |

## 決策

**選擇: 選項C - 單一 Repo，多專案目錄**

採用以下目錄結構：
```
gold-analysis/
├── backend/app/          # core（核心）
├── frontend/src/         # core（前端）
├── docs/                 # 文檔
├── extend/               # 擴展功能（待開發）
├── advanced/             # 進階功能（待開發）
└── platform/             # 平台功能（待開發）
```

## 依賴關係

```
platform/T003 依賴 extend/T005（多語言支持）
extend/* 依賴 core/*
advanced/* 依賴 core/*
```

## 執行順序

1. **第一波**: extend + advanced + platform/T001+T002 並行
2. **第二波**: platform/T003（等待 extend/T005 完成）

## 影響

### 正面影響
- 團隊協作順暢
- 依賴管理清晰
- 可分階段交付

### 負面影響
- 需要嚴格遵守目錄規範
- Git history 會較複雜
